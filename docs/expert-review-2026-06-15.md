# Kokoro TTS 专家小组评审纪要

日期：2026-06-15

## 专家分工

1. TTS/音频工程：分段、边界平滑、采样率与音质风险。
2. 后端/API：生命周期、并发、错误契约与端口协议。
3. 浏览器 UX：选择文本、取消、播放状态、资源清理与无障碍。
4. Windows/DevOps：Conda、安装脚本、托盘进程、日志与分发。
5. 性能/GPU：冷启动、显存、首包延迟、吞吐与基准方法。
6. 安全/隐私：loopback 边界、参数校验、DoS、CORS 与供应链。
7. 产品/开源：首次成功路径、文档可信度与功能取舍。
8. QA/可靠性：模型替身、API 契约、进程边界与回归矩阵。

## 一致结论

参考报告提出的模型预热、分段静音、淡入淡出和测试建设方向成立，
但首轮优先级必须先覆盖当前可复现的产品契约缺陷：

- 端口占用后静默切换会让浏览器和托盘连接错误服务。
- 托盘保存的 voice/speed 没有传递给服务器。
- 用户取消界面后，旧请求仍可能完成并突然播放。
- 托盘不读取子进程管道，长期运行可能阻塞。
- 安装失败后仍可能显示 Setup complete。
- voice、speed 和未知请求字段缺少严格校验。

## 本轮已实施

- 固定 API 端口，冲突时明确失败。
- `/health` 增加服务身份、版本、PID、ready 和默认配置。
- 托盘使用 `/health` 验证身份，配置通过环境变量传入服务。
- 托盘日志写入 `%LOCALAPPDATA%\KokoroTTS\server.log`。
- 安装脚本逐步检查失败，使用目标 Conda 环境执行 pip。
- 启动时执行默认声音 warmup。
- 分段静音和淡入淡出保留为可配置实验项，基于实测默认关闭。
- 英式声音使用 British English G2P pipeline，避免仅更换 voice tensor。
- 严格验证声音、语速、文本长度和未知字段。
- 忙碌时快速返回 429，避免无界等待。
- 用户脚本支持取消请求、全局设置存储和完整 Blob URL 清理。
- 增加不加载真实 Kokoro/CUDA 的音频与 API 自动化测试。

## 暂不实施

- 伪流式 WAV：逐段发送独立 RIFF/WAV 文件不是有效连续音频流。
- OGG/Opus：localhost 短文本传输不是当前瓶颈。
- FP16/torch.compile：必须先建立音质、显存和延迟基准。
- Docker、模型适配层、多语言、历史记录：偏离当前英文划词朗读主线。
